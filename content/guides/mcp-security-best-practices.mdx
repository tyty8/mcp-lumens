## Why Security Matters for MCP Servers

MCP servers are not passive APIs — they are agents that can read files, execute database queries, post to communication tools, and browse the web on behalf of an AI. A misconfigured server can expose sensitive data, allow unintended mutations, or become an attack vector if inputs are not validated.

Before deploying any MCP server beyond your local machine, work through the five practices below.

---

## 1. Principle of Least Privilege

Grant each server only the permissions it needs to do its job. Do not give a server access to resources it does not use.

**Filesystem:** Restrict the allowed root paths to the specific directories the AI should touch. Never point the filesystem server at `/` or `C:\`.

```json
{
  "args": [
    "-y",
    "@modelcontextprotocol/server-filesystem",
    "/home/app/data"
  ]
}
```

**Databases:** Create a dedicated database user with the minimum required privileges. For read-only analysis, use a `SELECT`-only role:

```sql
CREATE USER mcp_readonly WITH PASSWORD 'strongpassword';
GRANT CONNECT ON DATABASE mydb TO mcp_readonly;
GRANT USAGE ON SCHEMA public TO mcp_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO mcp_readonly;
```

Then connect with that user instead of your application's write credentials.

---

## 2. Authentication and API Keys

MCP servers often require API keys or tokens to access external services. Mishandling these credentials is the most common security mistake.

- **Never hardcode credentials** in config files committed to source control.
- **Use environment variables** and load them from a secrets manager or `.env` file that is excluded from git.
- **Rotate keys regularly.** Treat any key stored in a config file as potentially leaked.
- **Use scoped tokens.** GitHub Personal Access Tokens support fine-grained permissions — create one per server with only the scopes that server needs.

In Claude Desktop config, pass secrets through `env`:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

---

## 3. Input Validation

AI-generated inputs flow directly into your servers as tool arguments. Treat them with the same skepticism you would apply to user-supplied input in a web application.

- **Sanitize file paths.** Check that resolved paths stay within the allowed root before performing any file operation. Reject paths containing `..` traversal sequences.
- **Parameterize SQL queries.** Never interpolate AI-provided strings directly into SQL. Use parameterized queries or prepared statements to prevent injection.
- **Validate against a schema.** The MCP TypeScript SDK's Zod integration enforces input types at the boundary. Define strict schemas for every tool argument.

```ts
server.tool(
  "read_report",
  "Read a report file by name",
  {
    filename: z
      .string()
      .regex(/^[a-zA-Z0-9_-]+\.csv$/, "Invalid filename format"),
  },
  async ({ filename }) => {
    // filename is guaranteed to match the safe pattern
    const fullPath = path.join(REPORTS_DIR, filename);
    // ...
  }
);
```

---

## 4. Network Security

By default, MCP servers communicate over stdio with the local client. If you expose a server over a network (for remote agents or multi-user setups), additional controls are required.

- **Localhost by default.** Keep servers bound to `127.0.0.1` unless you explicitly need remote access.
- **Use TLS for remote transport.** Never send MCP traffic over plain HTTP on a network you do not fully control.
- **Firewall rules.** If running on a server, restrict inbound connections to trusted IP ranges.
- **Avoid exposing admin interfaces.** Do not run MCP servers with database superuser credentials or filesystem access to system directories on internet-facing machines.

---

## 5. Audit Logging

You should be able to answer: what did the AI do, when, and with what inputs? Without logging, anomalies are invisible until damage is done.

- **Log every tool invocation.** Record the tool name, input arguments, and outcome with a timestamp.
- **Centralize logs.** Ship logs to a central store (Datadog, CloudWatch, Loki) rather than writing only to local disk.
- **Alert on anomalies.** Set alerts for unusual patterns — bulk file reads, large SQL result sets, messages sent to unexpected Slack channels.
- **Retain logs.** Keep at minimum 30 days of audit logs for incident investigation.

A simple logging wrapper in a custom server:

```ts
server.tool("query_db", "Run a SQL query", { sql: z.string() }, async ({ sql }) => {
  console.log(JSON.stringify({
    event: "tool_invocation",
    tool: "query_db",
    input: { sql },
    timestamp: new Date().toISOString(),
  }));

  const result = await runQuery(sql);

  console.log(JSON.stringify({
    event: "tool_result",
    tool: "query_db",
    rowCount: result.rows.length,
    timestamp: new Date().toISOString(),
  }));

  return { content: [{ type: "text", text: JSON.stringify(result.rows) }] };
});
```

## Example: Secure PostgreSQL Configuration

Combining the practices above into a production-ready Postgres server setup:

```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "postgresql://mcp_readonly:${DB_PASSWORD}@db.internal:5432/analytics"
      ],
      "env": {
        "DB_PASSWORD": "${DB_PASSWORD}"
      }
    }
  }
}
```

- Uses a `mcp_readonly` role with `SELECT`-only grants
- Password loaded from an environment variable, not hardcoded
- Connects to an internal hostname, not a public endpoint
- Pair with audit logging in your database (pg_audit extension) for full observability
